# /sdd:patterns — 设计模式选型参考（24 种）

内置 24 种软件设计模式（GoF 23 + 简单工厂），每种标注**适用场景 / 解决什么问题 / 何时慎用**。用于 `/sdd:plan` 设计选型、`design-critic` 评审是否过度/恰当。

## 用户输入
$ARGUMENTS

## 行为
- **空参** → 输出下方分类总表。
- **给出问题/场景描述** → 从 24 种里挑**最贴切的 1–3 个**，说明为何适用、怎么落地，并**指出更简单的替代或反模式风险**（很多问题根本不需要模式）。
- **给出模式名** → 详述该模式：意图、场景、解决的问题、典型实现、慎用点。

> ⚠️ 头号原则（YAGNI）：**先问"这问题真需要模式吗"**。简单问题直接写最清晰；模式是用来驯服"反复出现的、会变化的复杂度"，不是用来炫技。为用而用 = 过度设计。

## 创建型 / Creational（6）——"怎么造对象"
| 模式 | 适用场景 | 解决什么问题 | 慎用 |
|------|---------|-------------|------|
| **简单工厂 Simple Factory** | 产品种类少且固定，创建逻辑简单 | 把分散的 `new` 集中到一处，客户端不依赖具体类 | 新增产品要改工厂（违反开闭）；种类多就升级工厂方法 |
| **工厂方法 Factory Method** | 产品会扩展，想让子类决定造哪个 | 把实例化延迟到子类，新增产品不改旧代码（开闭） | 每加一个产品多一个工厂类，类数膨胀 |
| **抽象工厂 Abstract Factory** | 要成套创建相关对象（跨平台 UI 套件、不同数据库驱动族） | 保证同族产品搭配一致、隔离具体实现 | 新增"产品等级"要改所有工厂；族不稳定时别用 |
| **建造者 Builder** | 复杂对象、可选参数多、构建分步骤（复杂配置、查询/请求 builder） | 替代重叠构造函数，让构建过程可控可读 | 简单对象用它是过度设计 |
| **原型 Prototype** | 创建成本高、需大量相似对象、想跳过重复初始化 | 用 clone 代替 new | 深/浅拷贝、循环引用易出错 |
| **单例 Singleton** | 全局唯一资源（配置、连接池、日志器） | 控制只有一个实例 + 全局访问点 | 滥用=全局状态，难测试；并发要保证线程安全 |

## 结构型 / Structural（7）——"怎么组合类与对象"
| 模式 | 适用场景 | 解决什么问题 | 慎用 |
|------|---------|-------------|------|
| **适配器 Adapter** | 复用接口不兼容的现有/第三方/遗留类 | 把已有接口转成客户端期望的接口 | 适配器满天飞 = 接口设计本身有问题 |
| **桥接 Bridge** | 两个维度都会独立变化（形状×颜色、消息类型×发送渠道） | 抽象与实现解耦，避免继承的笛卡尔积爆炸 | 维度不会独立变化时是多余抽象 |
| **组合 Composite** | 树形结构（文件系统、菜单、组织架构、UI 树） | 让客户端统一对待"单个"与"组合"对象 | 类型约束变弱，可能误用 |
| **装饰器 Decorator** | 运行时灵活叠加功能（IO 流、加缓存/日志/限流/鉴权层） | 替代子类爆炸，动态组合职责 | 多层装饰调试困难、顺序敏感 |
| **外观 Facade** | 子系统复杂，想给客户端一个简单入口 | 降低耦合、简化使用 | 容易膨胀成"万能类" |
| **享元 Flyweight** | 海量细粒度相似对象（字符、棋子、图标、粒子） | 共享内部状态，降内存占用 | 内/外部状态拆分复杂；非内存瓶颈别用 |
| **代理 Proxy** | 延迟加载/访问控制/远程调用/缓存（虚拟、保护、远程代理） | 不改对象的前提下控制对它的访问 | 多一层间接，注意性能与复杂度 |

## 行为型 / Behavioral（11）——"对象间怎么协作与分配职责"
| 模式 | 适用场景 | 解决什么问题 | 慎用 |
|------|---------|-------------|------|
| **责任链 Chain of Responsibility** | 请求需多级处理且处理者可变（审批流、中间件管道、过滤器） | 解耦发送者与处理者，动态编排处理链 | 链太长难调试；可能无人处理 |
| **命令 Command** | 需要撤销/重做、任务队列、宏、事务（编辑器、调度） | 把请求封装成对象，支持参数化/排队/undo | 简单直接调用用它是过度 |
| **解释器 Interpreter** | 解释一种简单自定义语言/表达式（规则引擎、DSL、计算器） | 为文法建模并求值 | 文法复杂就难维护，优先用现成解析器 |
| **迭代器 Iterator** | 遍历集合而不暴露内部结构、需自定义遍历 | 统一遍历接口、隐藏内部表示 | 现代语言多已内置，少自造 |
| **中介者 Mediator** | 多对象多对多复杂交互（聊天室、表单组件协调、塔台） | 用中介集中交互，降低对象间耦合 | 中介者易膨胀成"上帝对象" |
| **备忘录 Memento** | 撤销/快照/存档（编辑器历史、游戏存档） | 不破坏封装地保存/恢复状态 | 大状态内存开销大 |
| **观察者 Observer** | 事件订阅、状态变化驱动多方（MVC、发布订阅、响应式 UI） | 解耦发布者与订阅者，一对多通知 | 通知链复杂；忘记退订致内存泄漏 |
| **状态 State** | 行为随内部状态变且转换复杂（订单/连接状态机、游戏角色） | 用状态对象消除大量 if/switch 分支 | 状态很少时是过度设计 |
| **策略 Strategy** | 同一任务多种可换算法/规则（支付方式、排序、折扣、校验） | 算法可互换、运行时切换、消除条件分支 | 策略过多增加管理成本 |
| **模板方法 Template Method** | 多个流程骨架相同、个别步骤不同（框架钩子、流程模板） | 复用算法骨架，把变化点下放子类 | 继承耦合、灵活性不如策略 |
| **访问者 Visitor** | 对象结构稳定、却要频繁新增作用于它的操作（AST/编译器、报表） | 操作与结构分离，加新操作不改元素类 | 元素类频繁变动则灾难（要改所有 visitor） |

## 选型速查（按"我遇到的问题"找）
- **到处 `new`、创建逻辑散乱** → 简单工厂 / 工厂方法 / 抽象工厂（按扩展性递增）
- **构造参数爆炸** → 建造者
- **一堆 if/else 选算法或规则** → 策略；**一堆 if/else 随状态切行为** → 状态
- **想给对象动态加功能** → 装饰器；**想控制对它的访问** → 代理
- **接口对不上 / 接旧系统** → 适配器
- **两个维度组合爆炸** → 桥接
- **状态变化要通知一群人** → 观察者
- **要撤销/重做/任务队列** → 命令(+备忘录)
- **多级处理一个请求** → 责任链
- **树形结构统一处理** → 组合
- **海量重复小对象吃内存** → 享元
- **子系统太复杂想要简单入口** → 外观

## 纪律
- ✅ 先判断"是否真需要模式"——简单清晰 > 模式正确。
- ✅ 选用时结合 constitution 的架构原则与激活的能力包约定；遵守分层/依赖方向。
- ❌ 不为展示而堆模式；过度设计会被 `design-critic` 和可维护性门禁打回。
