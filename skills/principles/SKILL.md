# /sdd:principles — 工程原则参考（设计 / 架构 / 云原生 / 高可用）

内置四组工程原则，每条标**解决什么问题**（部分标"何时用/慎用"）。与 `/sdd:patterns` 互补：**原则定方向，模式给招式**。供 `/sdd:plan` 选型、`design-critic` 评审参考。

## 用户输入
$ARGUMENTS

## 行为
- **空参** → 输出下方四组总表。
- **给出问题/场景** → 挑**最贴切的几条**，说明为何适用、怎么落地，并指出反模式/过度风险。
- **给出原则名** → 详解：含义、解决的问题、落地要点、慎用点。

> ⚠️ 原则是方向盘不是教条：结合上下文取舍，别 cargo-cult。原则之间也会张力（如 DRY vs 解耦、抽象 vs KISS）——优先**简单清晰**，复杂度只在真有变化压力时引入。

## A. 设计原则（类/模块层）
| 原则 | 解决什么问题 |
|------|-------------|
| **SRP 单一职责** | 一个类只有一个变化的理由 → 改动波及面小 |
| **OCP 开闭原则** | 对扩展开放、对修改关闭 → 加功能不动旧代码 |
| **LSP 里氏替换** | 子类能无痛替换父类 → 继承不埋雷 |
| **ISP 接口隔离** | 不强迫依赖用不到的方法 → 接口小而专 |
| **DIP 依赖倒置** | 依赖抽象而非具体 → 解耦、可测、可替换 |
| **DRY 不重复** | 消除重复的"知识" → 改一处不遗漏（治冗余） |
| **KISS 保持简单** | 怎么简单怎么来 → 压低复杂度 |
| **YAGNI 不做没用的** | 不提前造想象功能 → 防过度设计 |
| **高内聚低耦合** | 相关的聚一起、无关的分开 → 易改易测 |
| **组合优于继承** | 组合替代深继承 → 灵活、避免脆弱基类 |
| **迪米特法则（最少知识）** | 只跟直接朋友打交道 → 降耦合 |
| **关注点分离 SoC** | 不同职责拆开 → 结构清晰 |
| **单一事实来源 SSOT** | 一份权威数据/定义 → 防不一致 |
| **营地法则 Boy Scout Rule** | 离开时让代码比来时更干净 → 持续抗腐化 |

## B. 架构方法（系统层）
| 方法 | 解决什么问题 | 何时用 |
|------|-------------|--------|
| **整洁架构 Clean Architecture** | 依赖只向内指、业务核心不依赖框架/DB/UI → 核心稳定可测、外围可替换 | 业务逻辑重、生命周期长的系统 |
| **六边形（端口与适配器）** | 业务经端口与外界交互 → 隔离外部依赖、易测 | 要换外设/重测试的服务 |
| **分层架构 Layered** | 表现/业务/数据分层 → 职责清晰 | 大多数常规应用的默认骨架 |
| **DDD 领域驱动设计** | 围绕领域用统一语言建模 → 驾驭复杂业务 | 业务复杂、规则多变；简单 CRUD 别上 |
| **CQRS 读写分离** | 读写模型分开 → 各自优化 | 读写负载/模型差异大时；否则过度 |
| **事件驱动 / 事件溯源** | 用事件解耦 → 异步、可追溯、可重放 | 高解耦、审计/回放需求；增加最终一致复杂度 |

## C. 云原生 / 12-Factor App
> 让应用可移植、易弹性伸缩、利于云部署。
1. **Codebase** 一份代码多处部署（版本控制）
2. **Dependencies** 显式声明并隔离依赖
3. **Config** 配置存于环境变量（不写死进代码）
4. **Backing services** 后端服务（DB/队列/缓存）当可插拔附加资源
5. **Build/Release/Run** 三阶段严格分离
6. **Processes** 进程无状态、无共享
7. **Port binding** 自带服务、端口绑定暴露
8. **Concurrency** 用进程模型水平扩展
9. **Disposability** 快速启动、优雅关闭
10. **Dev/prod parity** 开发/生产环境尽量一致
11. **Logs** 日志当事件流（输出到 stdout，不自管文件）
12. **Admin processes** 管理任务作为一次性进程

## D. 可靠性 / 高可用
| 原则 | 解决什么问题 | 何时用 |
|------|-------------|--------|
| **失败快速 Fail Fast** | 错误早暴露，不带病运行 | 输入校验、启动自检 |
| **幂等 Idempotency** | 重复执行结果一致 → 安全重试 | 支付/下单/消息消费等写操作 |
| **超时 Timeout** | 别无限等待拖垮自己 | 所有外部调用 |
| **重试 + 退避 Retry/Backoff** | 扛瞬时故障 | 网络/依赖抖动；需配合幂等 |
| **熔断 Circuit Breaker** | 下游挂了快速失败 → 防雪崩 | 强依赖的远程调用 |
| **舱壁隔离 Bulkhead** | 隔离资源池 → 故障不扩散 | 多依赖共享线程/连接池 |
| **优雅降级 Graceful Degradation** | 部分失败仍保住核心功能 | 非核心依赖不可用时 |
| **可观测性 Observability** | 日志/指标/追踪 → 能定位问题 | 任何要运维的线上系统 |
| **防御式 / 契约式编程** | 校验输入、前/后置条件 → 健壮 | 模块边界、公共 API |

## 选型速查（按"我遇到的问题"找）
- **改一个功能牵动一片** → SRP / 高内聚低耦合 / 关注点分离
- **加功能总要改旧代码** → OCP / DIP（+ 策略/工厂模式）
- **代码重复、改不胜改** → DRY / SSOT（+ 复用发现）
- **过度抽象、看不懂** → KISS / YAGNI（做减法）
- **业务复杂、概念混乱** → DDD + Clean Architecture
- **核心逻辑被框架/DB 绑死、难测** → 整洁/六边形架构、DIP
- **上云部署不顺、配置乱** → 12-Factor
- **依赖一抖整个挂 / 要扛流量** → 超时+重试+熔断+舱壁+降级 + 可观测性

## 纪律
- ✅ 先简单后复杂；原则用来**驯服真实的变化压力**，不是装饰。
- ✅ 与 constitution 的架构原则、激活能力包、`/sdd:patterns` 协同。
- ❌ 不 cargo-cult、不为"显得专业"硬套架构——过度设计会被 `design-critic` 与可维护性门禁打回。
